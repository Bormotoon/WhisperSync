## TL;DR

Проект в хорошей инженерной форме: чистые линты и типы, зелёные тесты, продуманная архитектура engine/gui, отличная история отладки (баг кумулятивного дрейфа atempo найден и убит). Но за время экспериментов накопился существенный долг:

1. **Итоговый звук заметно хуже исходника по качеству.** Рендер-путь принудительно даунмиксит в моно, режет битность до 16 бит, роняет громкость в ноль на каждом шве и использует atempo/WSOLA там, где для микро-дрейфа хватило бы прозрачного ресемплинга. См. [§2.0](#20-деградация-качества-звука-в-рендер-пути-).
2. **Стратегии синхронизации больше не делают то, что обещают.** `strategies.py` (300 строк) мёртв в продакшене — пайплайн использует `clip_pieces()`, где стратегии 3 и 4 идентичны, а «pitch-safe Silence Padding» на самом деле тайм-стретчит речь.
3. **Заикание на швах (tempo-break stutter) не решено при дефолтах.** Фикс `smooth_tempo` признан вредным и выключен; настоящий фикс (seam-snap-to-silence) не реализован.
4. **Рендер-путь опасен для не-WAV рекордеров** (mp3/m4a: `-ss` до `-i` не сэмпло-точен) и для рекордеров с sample-rate ≠ выходного (concat разнородных WAV).
5. **Pause ducking может глушить настоящую речь** — паузы выводятся из якорей, а не из VAD.
6. Много мёртвого кода, рассинхрон README ↔ поведение, дефолты GUI ≠ CLI, нет ни одного теста, реально гоняющего ffmpeg (а худший исторический баг жил именно там).

Ниже — полный разбор с файлами/строками, решениями и приоритетами.

## 1. Сильные стороны

Чтобы отчёт не выглядел одной сплошной претензией — что сделано хорошо:

- **Чистое разделение engine/gui** — бизнес-логика не зависит от Qt, воркер через `moveToThread`, сигналы вместо прямых вызовов.
- **Дисциплина качества**: type hints везде, ruff/black/mypy зелёные, 74 юнит-теста на чистой логике без GPU/ffmpeg — быстрые и детерминированные.
- **Продуманный Whisper-движок**: OOM-лестница (батч → батч/2 → CPU), анти-галлюцинационная температурная лестница, кэш транскрипций по SHA-256, `HF_HUB_DISABLE_XET` workaround.
- **Сильная алгоритмика**: coarse-voting по редким словам для многочасовых рекордеров, RANSAC поверх локального отсева выбросов, GCC-PHAT с параболической субсэмпловой интерполяцией и sharpness-гейтом.
- **Победа над кумулятивным дрейфом**: `apad+atrim` в `apply_atempo_segment` (−2.97 мс/кусок → −0.007 мс) — образцовая отладка, зафиксированная регрессионным тестом.
- **Concat через demuxer**, а не тысячи `-i` (лимит файловых дескрипторов) — правильное решение.
- **Изоляция тяжёлой сепарации** в `.sep-venv` с вызовом через console-script — прагматично решён конфликт Python 3.14 / torch.

## 2. Критические проблемы (корректность)

### 2.0. Деградация качества звука в рендер-пути 🔴🔴

Это добавленная пользователем находка — итоговые монолитные `*_voice.wav` заметно хуже исходной записи рекордера на слух, хотя инструмент по духу должен сохранять качество звука неизменным (в проекте нет ни одного шага, который *обязан* терять качество: задача — сдвиг во времени, а не обработка сигнала). Пять независимых причин, каждая усиливает остальные:

1. **Принудительный даунмикс в моно.** `assemble_continuous` резэмплит и форсирует `aformat=channel_layouts=mono` + `-ac 1` ([timestretch.py:361-367](whispersync/engine/timestretch.py#L361-L367)); то же в `apply_pause_ducking` ([timestretch.py:90-93](whispersync/engine/timestretch.py#L90-L93)). Стереорекордер (Zoom H1/H5, Tascam, любая XY/ORTF пара) сплющивается в моно безвозвратно — теряется вся стереопанорама/ширина, которую пользователь наверняка хотел сохранить.
2. **Принудительные 16 бит на каждом шаге.** `extract_segment`, `apply_atempo_segment`, `generate_silence`, `apply_pause_ducking` — все пишут `-acodec pcm_s16le` ([timestretch.py:161](whispersync/engine/timestretch.py#L161), [:193](whispersync/engine/timestretch.py#L193), [:237](whispersync/engine/timestretch.py#L237), [:95](whispersync/engine/timestretch.py#L95)). Профессиональные рекордеры пишут 24-bit (иногда 32-bit float); каждое промежуточное round-trip через 16 бит без дизера — это не одно квантование, а *цепочка* квантований (кусок → атемпо → фейд → конкатенация → дакинг), потому что рендер многопроходный. Кумулятивный шум квантования на десятках/сотнях кусков слышен как лёгкое шипение/огрубление тембра.
3. **Провалы громкости на каждом шве.** `edge_fade_filters` ставит fade-in/out **на каждом куске** ([timestretch.py:16-30](whispersync/engine/timestretch.py#L16-L30), включено по умолчанию через `crossfade_enabled=True`), но куски стыкуются встык (`assemble_continuous` просто конкатенирует — не кроссфейдит соседние файлы). Fade-out одного куска до нуля + fade-in следующего от нуля на смежной границе = слышимая просадка громкости (по сути короткий гейт/затухание) на **каждом** шве, а не декликанье щелчка. Для стратегии 2 (кусок на каждый якорь) это происходит почти на каждом слове — ощущается как «рубленая», неестественно пульсирующая по громкости речь.
4. **WSOLA-артефакты atempo там, где они не нужны.** Реальный дрейф часов между камерой и рекордером — доли процента (K обычно 0.995–1.005). `ffmpeg atempo` — это WSOLA time-stretch, спроектированный для changes от ±many% и вносящий фазовые/текстурные артефакты на любом множителе, включая крошечные. Для дрейфа ≤0.3-0.5% практически неотличимый от исходника результат даёт простой ресемплинг (`asetrate`+`aresample`, эквивалент "varispeed") — питч сдвигается на те же <0.5% (< 9 центов, физически неразличимо на речи), но временная структура сигнала остаётся нетронутой, в отличие от WSOLA. Сейчас atempo применяется единообразно на любой величине фактора.
5. **Множественные полные decode/encode-проходы.** Кусок проходит: `extract`/`atempo` (кодирование #1) → `assemble_continuous` ресэмплинг+конкатенация (#2) → при включённом pause-duck ещё один полный проход через `apply_pause_ducking` поверх уже собранного `raw_*.wav` (#3, [pipeline.py:807-817](whispersync/engine/pipeline.py#L807-L817)). Каждый проход — это лишняя генерация/потеря (особенно ощутимо в сочетании с п.2).

Отдельно усугубляет: резка кусков из **оригинального** файла рекордера, если это lossy-формат (mp3/m4a — см. §2.3 ниже), означает, что каждый кусок уже проходит через lossy-декодирование заново при каждом `-ss`-сиде, и `extract_segment`/`apply_atempo_segment` дополнительно перекодируют результат в 16-bit PCM — то есть *исходное* lossy-сжатие остаётся, но поверх него добавляется ещё и деградация от многопроходного рендера.

**Итог:** ни один из этих пяти пунктов не даёт никакого выигрыша в синхронизации — это чистые потери качества, введённые случайно (моно/16-бит как "безопасный дефолт" для WAV-вывода, фейды скопированы из декликанья реальных стыков без учёта того, что новые швы искусственные и смежные). Все пять устраняются без ущерба для синхронизации: сохранить нативные каналы/битность рекордера сквозь весь пайплайн, ставить фейды только на реально несмежных/акустически подвинутых швах, использовать resample-conform вместо atempo при малых факторах, свернуть pause-ducking в тот же фильтр-граф, что и сборка (один проход вместо двух). План работ — Этап 1 в `~/.claude/plans/kind-baking-thacker.md`.

### 2.1. `strategies.py` мёртв; стратегии 3 и 4 идентичны; «Silence Padding» больше не pitch-safe 🔴

`run_pipeline` вызывает `get_strategy(strategy_id)` только ради `strategy.name` ([pipeline.py:639](whispersync/engine/pipeline.py#L639), [pipeline.py:762](whispersync/engine/pipeline.py#L762)). Реальное планирование делает `clip_pieces()` ([pipeline.py:261](whispersync/engine/pipeline.py#L261)), в котором:

- Стратегия 1 — один глобальный stretch;
- Стратегия 2 — кусок на каждый якорь;
- Стратегии **3 и 4 — один и тот же код** (`if strategy_id in (3, 4)`, [pipeline.py:313](whispersync/engine/pipeline.py#L313)): прореживание якорей до ~одного на фразу и **piecewise atempo-стретч речи**.

Последствия:

- Обещание стратегии 3 («речь не трогается, ноль pitch-артефактов» — README, GUI, help-страница, docstring `SilencePaddingStrategy`) **нарушено молча**: речь стретчится.
- Пользователь, выбирающий между 3 и 4, выбирает между двумя одинаковыми вещами.
- 300 строк `plan_clip`-логики в [strategies.py](whispersync/engine/strategies.py) не исполняются в проде, но прилежно тестируются в [test_strategies.py](tests/test_strategies.py) — тесты проверяют мёртвый код и создают ложное чувство покрытия.

**Решение (одно из двух):**
- **(а)** Вернуть честную стратегию 3 в `clip_pieces`: куски речи режутся *без* стретча (factor=1.0), позиционируются по якорям, зазоры заполняет `assemble_clip`-подобная укладка на тишину (код `assemble_clip` уже есть и тоже мёртв — см. §4). Это восстанавливает обещание pitch-safe.
- **(б)** Честно объединить 3 и 4 в одну стратегию «Phrase-wise», обновить README/GUI/симулятор, удалить `plan_clip` и его тесты.

### 2.2. Tempo-break stutter не решён при дефолтных настройках 🔴

Хронология (по коммитам и памяти проекта): заикание «подга-га-товил» на швах кусков → фикс `_smooth_piece_tempo` (сглаживание факторов) → обнаружено, что сглаживание **само уводит речь от картинки до ~1.4 с** (параболическая ошибка) → `smooth_tempo=False` по умолчанию ([config.py:144-149](whispersync/config.py#L144-L149)) с комментарием «настоящий фикс — seam-snap-to-silence». 

**Итог: сегодня при дефолтах воспроизводится исходная жалоба пользователя** — резкие скачки atempo-фактора между соседними кусками (в реальных данных до 40%) на швах внутри слова.

**Решение:** реализовать seam-snap-to-silence — сдвигать границы кусков (breakpoints в `clip_pieces`) к ближайшему меж-словному зазору транскрипта (границы слов уже есть в `Transcript`), чтобы шов никогда не падал внутрь слова. Факторы кусков не трогаются → синхронизация сохраняется точно. Дополнительно: ограничить скачок фактора *через перераспределение breakpoint'ов*, а не факторов. `_smooth_piece_tempo` после этого удалить (см. §4).

### 2.3. Не-WAV рекордер: резка кусков не сэмпло-точная 🔴

`audio_exts` допускает `.mp3`, `.m4a`, `.flac` ([config.py:14](whispersync/config.py#L14)). Куски режутся из **оригинального файла рекордера** командой `-ss <start> -t <dur> -i <file>` ([timestretch.py:149-163](whispersync/engine/timestretch.py#L149-L163), [pipeline.py:800](whispersync/engine/pipeline.py#L800)). Быстрый input-seek сэмпло-точен только для PCM/WAV; для mp3/m4a ffmpeg садится на границу фрейма (~±26 мс для mp3) плюс decoder priming/padding. Для сотен кусков это систематический джиттер границ ровно того масштаба (±1 кадр), с которым проект борется. Тот же дефект — в `extract_audio_window` для Boundary Flex ([media.py:144](whispersync/engine/media.py#L144)).

**Решение:** в начале пайплайна один раз транскодировать каждый рекордер в PCM WAV на выходном sample-rate (`out_sr`, mono) во временный файл и все резки/флекс-окна делать из него. Побочный бонус — решается и §2.4.

### 2.4. `assemble_continuous` конкатенирует разнородные WAV 🔴

Куски наследуют sample-rate и число каналов рекордера (в `extract_segment`/`apply_atempo_segment` нет `-ar`/`-ac`), а lead-тишина генерируется в `out_sr` mono ([timestretch.py:339-341](whispersync/engine/timestretch.py#L339-L341)). Concat-demuxer требует одинаковых параметров потоков; при рекордере 44.1 кГц/стерео и `out_sr=48000` склейка даёт сломанные таймстемпы/скорость до того, как `aresample` в `-af` успеет что-то исправить. Сейчас это спит, потому что типичный кейс — 48k рекордер и 48k камера.

**Решение:** форсировать `-ar {out_sr} -ac 1` во всех рендерах кусков (`extract_segment`, `apply_atempo_segment`), либо решить через §2.3 (резать из уже нормализованного WAV).

### 2.5. Pause ducking глушит то, что не смогло стать якорем 🟠

`pause_spans_local` ([pipeline.py:385-412](whispersync/engine/pipeline.py#L385-L412)) считает «паузой» любой зазор между *якорями* > 0.6 с, включая head/tail клипа. Но якорь — это слово, которое (а) распозналось на обеих дорожках, (б) с probability ≥ 0.6, (в) совпало и пережило фильтры. Тихая реплика, слово с низким конфиденсом, кусок речи, который Whisper пропустил, приветствие до первого совпавшего слова — всё это будет приглушено на −18 дБ как «пауза».

**Решение:** строить паузы из **транскрипта рекордера** (все слова, без порога уверенности, спроецированные в локальное время), а лучше — пересечь с VAD-интервалами (Silero VAD уже прогоняется внутри faster-whisper; параметр `vad_filter=True`). Дакать только там, где *обе* дорожки молчат.

### 2.6. Фильтр грубых выбросов отключается на коротких клипах 🟠

`reject_gross_outliers` возвращает всё как есть при `len(anchors) < 2*window = 20` ([matcher.py:161](whispersync/engine/matcher.py#L161)). Короткий клип с 10–19 якорями и одним выбросом в 5 секунд получит искорёженный piecewise-варп. Кроме того, память проекта фиксирует, что одиночные выбросы с резидуалом 0.5–5.5 с выживали и при полном окне. `window`/`tol_s` захардкожены и не вынесены в конфиг.

**Решение:** при малом числе якорей уменьшать окно (например, `window = max(2, len(anchors)//4)`), а не отключать фильтр; добавить второй проход — отбрасывать якоря с резидуалом против RANSAC-прямой > N×median (изолированные одиночки); вынести параметры в конфиг.

### 2.7. Кэш транскрипций: ключ не учитывает фактический device/compute_type 🟠

`_cache_key` включает `config.compute_type` — часто это строка `"auto"` ([transcriber.py:265-289](whispersync/engine/transcriber.py#L265-L289)). Фактический прогон может быть cuda-float16, а после OOM-деградации — cpu-float32; результаты различаются, но лягут под тем же ключом, и наоборот — смена GPU→CPU не инвалидирует кэш. Также в ключе нет `batch_size`-независимости проблемы нет, но нет и версии faster-whisper/модели весов.

**Решение:** класть в ключ *резолвнутые* `self._device`/`self._compute_type` (известны в момент вызова `transcribe`) + версию faster-whisper.

### 2.8. Тихая потеря видео в корне при мультикаме 🟡

`scan_cameras`: если в `video_dir` есть и подпапки с видео, и «свободные» видеофайлы в корне — корневые файлы молча игнорируются ([pipeline.py:136-144](whispersync/engine/pipeline.py#L136-L144)).

**Решение:** warning в `warnings` (`"N video file(s) in the root of --video-dir ignored because camera sub-folders exist"`).

### 2.9. `load_config` молча игнорирует несуществующий `--config` и опечатки в ключах 🟡

`if config_path and config_path.exists()` ([config.py:224-230](whispersync/config.py#L224-L230)) — опечатка в пути = дефолтный конфиг без единого слова. `from_file` отбрасывает неизвестные ключи молча — опечатка `"pause_duck_dB"` просто не сработает.

**Решение:** несуществующий путь → ошибка; неизвестные ключи → warning со списком.

### 2.10. Placement по слабым alignment'ам без гейта 🟡

`_add_job`/`compute_master_offsets` принимают alignment с любым числом якорей ≥ 2. RANSAC-прямая по 2–3 якорям — это не регрессия, а лотерея; клип получит K и позицию из шума. Единственная защита — `sequence_order_warnings` постфактум.

**Решение:** для placement требовать `len(anchors) >= min_anchors` (уже есть в конфиге, но фактически используется только для warning'а); слабые клипы честно укладывать по порядку имён с warning'ом, как уже делается для полностью невыровненных.

## 3. Ошибки надёжности и кроссплатформенности

### 3.1. Windows фактически не поддерживается, вопреки README

- `path_to_file_uri` строит URI вручную: на Windows `str(path)` даёт backslashes, которые заквотятся в `%5C` → битый `file://` URI ([media.py:185-188](whispersync/engine/media.py#L185-L188)). Правильно: `Path.as_uri()`.
- «Open Output Folder» жёстко вызывает `xdg-open` ([main_window.py:501](whispersync/gui/main_window.py#L501)) — на Windows/macOS кнопка мертва. Правильно: `QDesktopServices.openUrl(QUrl.fromLocalFile(...))`.
- README обещает «Windows | macOS | Linux» в бейдже и таблице требований.

### 3.2. Относительный `media-rep src` без URL-энкодинга

`_media_src` для файлов рядом с FCPXML возвращает голый относительный путь ([export.py:21-32](whispersync/engine/export.py#L21-L32)). Имя клипа с пробелом/скобками (`My Clip (2)_voice.wav`) попадёт в атрибут `src` без percent-encoding — часть парсеров URI это отвергнет. Абсолютная ветка квотится, относительная — нет.

**Решение:** `quote(str(rel_path), safe="/")` и для относительной ветки.

### 3.3. `fork`-пул из GUI-потока с загруженной CUDA-моделью

`_pool_context()` предпочитает `fork` ([pipeline.py:422-434](whispersync/engine/pipeline.py#L422-L434), [acoustic.py:39-45](whispersync/engine/acoustic.py#L39-L45)). В GUI-режиме это fork многопоточного Qt-процесса с инициализированным CUDA-контекстом (ctranslate2) — классический источник дедлоков (форк захватывает состояние чужих локов malloc/logging/CUDA). Python ≥3.12 явно предупреждает об этом, поэтому 3.14 и сменил дефолт. Сейчас работает, потому что дети быстро уходят в `subprocess.run(ffmpeg)`, но это мина.

**Решение:** выгружать Whisper-модель до рендера (см. §6.1 — нужно и по VRAM), а ещё лучше — использовать fork только когда `threading.active_count() == 1`, иначе spawn. Обоснование скорости fork («короткие задачи») ослабнет, если убрать пер-оконный спавн ffmpeg (см. §6.2).

### 3.4. Видео без аудиодорожки роняет весь пайплайн

`extract_audio_to_wav` на видео без звука → `RuntimeError` → падение всего прогона ([pipeline.py:576](whispersync/engine/pipeline.py#L576)). Клип без звука (таймлапс, б-ролл) — обычное дело.

**Решение:** `probe()` уже знает `audio_codec is None` — такие клипы помечать «placed by order», предупреждать и пропускать транскрипцию.

### 3.5. Отмена не прерывает подпроцессы

Флаг отмены проверяется только в progress-callback ([worker.py:35-37](whispersync/gui/worker.py#L35-L37)). Во время транскрипции колбэки идут часто — ок; но во время рендер-джоба с тысячами кусков, `assemble_continuous` (минуты) или сепарации (минуты на клип) отмена не срабатывает до конца шага, а уже запущенные ffmpeg/audio-separator никто не убивает. `closeEvent` ждёт 3 с и закрывает окно с живым потоком — при выходе возможен crash.

**Решение:** передавать в пайплайн `cancel_event`; проверять его в циклах кусков/джобов; хранить `Popen`-хэндлы и `terminate()` их при отмене; в `closeEvent` не закрываться, пока поток жив (или detach с предупреждением).

### 3.6. `system_check.py` проверяет не то, чем пользуется приложение

Проверяет `torch.cuda` ([system_check.py:44-67](whispersync/engine/system_check.py#L44-L67)), но torch в основном venv **не установлен** (транскрипция идёт через ctranslate2) → пользователь с рабочим GPU видит «CUDA check skipped». Не проверяются: CUDA через `ctranslate2.get_cuda_device_count()` (как это делает боевой [resolve_device](whispersync/engine/transcriber.py#L52)), наличие `.sep-venv` для ambience, `report.json` пишется внутрь пакета (`whispersync/engine/report.json`), а README обещает его в корне.

Это главный барьер для интеграционного тестирования и для отзывчивой отмены.

**Решение:** разбить на стадии с явными вход/выход (dataclass `PipelineState`): `scan → transcribe → align → place → plan → render → ambience → export`. Каждая стадия тестируема отдельно; отмена проверяется между стадиями и внутри циклов.

### 4.3. Дублирование dry-run

### 4.4. Два источника дефолтов: GUI ≠ CLI ≠ config

| Параметр | config/CLI | GUI |
|---|---|---|
| Стратегия | `1` (`default_strategy`, [cli.py:134](whispersync/cli.py#L134)) | `4` checked ([main_window.py:138](whispersync/gui/main_window.py#L138)) |

README при этом называет стратегию 4 «рекомендуемой», а CLI-дефолт — 1. Пользователь CLI и пользователь GUI получают разные результаты на одинаковых данных.

**Решение:** один источник правды — `WhisperSyncConfig` (поднять дефолты `default_strategy=4`, `boundary_flex=True`, раз они «recommended/best out of the box»), GUI и CLI читают оттуда. `config.default_strategy`, кстати, сейчас не используется ни GUI, ни CLI.

### 4.5. GUI разделяет мутируемый config с воркером

`self.config` мутируется в `_start_sync` и передаётся воркеру по ссылке ([main_window.py:407-415](whispersync/gui/main_window.py#L407-L415)); изменение галочек во время прогона меняет объект под ногами пайплайна. Сейчас пайплайн читает поля в начале — но это хрупко. Решение: `dataclasses.replace(...)`/копия на старте.

## 6. Бутылочные горлышки производительности

### 6.1. Whisper-модель держит VRAM весь прогон

`engine.unload()` вызывается только в `finally` ([pipeline.py:931-933](whispersync/engine/pipeline.py#L931-L933)). После фазы выравнивания модель (~3–5 GB VRAM для large-v3/float16) не нужна, но остаётся загруженной во время рендера и — главное — во время ambience-сепарации, которая поднимает **свою** GPU-модель (1.57 GB ckpt) в `.sep-venv`. На 8–12 GB GPU это готовый OOM.

**Решение:** `engine.unload(); engine = None` сразу после цикла выравнивания.

### 6.2. Boundary Flex: два спавна ffmpeg на каждую границу

`_measure_boundary` пишет два маленьких WAV через два `subprocess.run(ffmpeg)` на **каждый** кусок ([acoustic.py:284-321](whispersync/engine/acoustic.py#L284-L321)). Для клипа на 784 куска — ~1600 процессов, издержки запуска доминируют над FFT.

**Решение:** прочитать оба трека один раз в память (mono 16k float: 93-минутный рекордер ≈ 178 MB — приемлемо; можно `np.memmap` через один предварительный ffmpeg-транскод) и резать окна слайсами numpy. Ожидаемое ускорение 10–50×, заодно исчезает mp3-seek проблема §2.3 для флекса. Параллелить тогда можно потоками (numpy FFT отпускает GIL) — снимается и fork-вопрос §3.3 для этого пула.

### 6.3. Ambience: загрузка 1.57 GB модели на каждый клип

`extract_ambience` вызывает CLI `audio-separator` **по одному файлу** ([pipeline.py:846-872](whispersync/engine/pipeline.py#L846-L872)); каждый вызов заново грузит checkpoint в GPU. 8 клипов → 8 полных загрузок (десятки секунд каждая).

**Решение:** `audio-separator` принимает несколько входных файлов за вызов (или папку) — собрать все camera-WAV во временную папку и сделать один вызов; либо держать резидентный процесс-воркер в `.sep-venv` с простым stdin-протоколом.

### 6.4. Рендер-джобы последовательны; ducking — лишний полный проход

- Параллелизм существует только *внутри* джоба (`render_pieces`); сами джобы идут по одному ([pipeline.py:770](whispersync/engine/pipeline.py#L770)). Много коротких клипов с малым числом кусков почти не параллелятся; `assemble_continuous` — серийный хвост каждого джоба.
- При включённом ducking результат сначала собирается в `raw_*.wav`, затем **полностью** перекодируется вторым ffmpeg-проходом ([pipeline.py:807-817](whispersync/engine/pipeline.py#L807-L817)). `volume`-фильтры можно дописать в `-af`-цепочку самого `assemble_continuous` — минус один полный decode/encode на клип.

### 6.5. Матчинг: повторная индексация рекордера на каждый клип

`estimate_coarse_delta` строит `Counter` и позиции по **всем словам рекордера** заново для каждой пары клип×рекордер ([matcher.py:51-58](whispersync/engine/matcher.py#L51-L58)); ретрай `align` на полном референсе ([matcher.py:285-287](whispersync/engine/matcher.py#L285-L287)) запускает `difflib.SequenceMatcher` O(n·m) на 90-минутном транскрипте — на слабом матче это десятки секунд на клип. Кэшировать индекс рекордера (`rec_positions`, нормализованные слова) один раз на рекордер.

### 6.6. Мелочи

- `probe()` timeout 30 с ([media.py:40](whispersync/engine/media.py#L40)) — MXF на сетевом хранилище может не уложиться; сделать конфигурируемым.
- Транскрипция клипов камеры идёт без progress-callback ([pipeline.py:578](whispersync/engine/pipeline.py#L578)) — на длинном клипе прогресс-бар замирает (это UX, но живёт в том же цикле).
- Кэш `~/.cache/whispersync/` растёт бесконечно (JSON транскрипта 93-минутной записи — десятки MB); добавить прунинг по возрасту/размеру.

## 7. UX-проблемы (GUI и CLI)

### GUI

1. **Один рекордер.** CLI умеет несколько `--audio-file` и `--recorder-mode`, GUI — только один файл. Дроп нескольких файлов должен собирать список.
2. **Нет настроек движка.** Модель/язык/устройство/`initial_prompt`/`--mode` недоступны — только дефолт large-v3+auto. Settings-диалог значится в roadmap (Phase 5) как несделанный.
3. **Нет индикации первой загрузки модели** (~3 GB с HuggingFace): окно выглядит зависшим на «Transcribing…». Нужен статус «Downloading model…» (faster-whisper даёт колбэки скачивания через huggingface_hub).
4. **Ambience-чекбокс активен даже без `.sep-venv`** — пользователь узнаёт о проблеме только warning'ом в конце. `separation.is_available()` дешев — дизейблить чекбокс с тултипом.
5. **Прогресс скачет**: каждая стадия сбрасывает бар 0→100 ([worker.py:39](whispersync/gui/worker.py#L39)), общий прогресс прогона не виден. Взвесить стадии (scan 2% / transcribe 60% / render 30% / export 8%).
6. **Warnings из результата не показываются** в `_on_finished` ([main_window.py:466-487](whispersync/gui/main_window.py#L466-L487)) — а там живут важные вещи («not aligned to any recorder», «High residual», «FCPXML failed validation»). CLI их печатает, GUI — нет.
7. **Логи engine-модулей не попадают в LogView**: GUI не вызывает `setup_logging`, а LogView получает только `PipelineProgress.message`. INFO-логи матчера (число якорей, K, residual per clip) — самое интересное — видны только в терминале.
8. Повторные прогоны создают новые `QThread`/`SyncWorker` без `deleteLater` — утечка мелкая, но есть.

### CLI

9. **`--json` испорчен прогрессом**: `_progress_printer` пишет в stdout вперемешку с финальным JSON ([cli.py:38-41](whispersync/cli.py#L38-L41)) — `whispersync ... --json | jq` не работает. Прогресс → stderr (или `--quiet`).
10. **Нет `--version`.**
11. **Нет `--strategy`-имени** (`--strategy hybrid`) и нет способа выключить `smooth_tempo`/включить `align_mode` из CLI (только JSON-конфиг) — осознанно ли?
12. Один код возврата на все ошибки (1); для автоматизации полезно различать «нет якорей» / «нет ffmpeg» / «плохие аргументы».

## 8. Тесты и CI

1. **Ноль тестов, исполняющих ffmpeg.** Вся сюита — чистая логика (это хорошо для скорости), но худший баг проекта (−2.97 мс/кусок) жил именно в ffmpeg-фильтрах и был бы пойман только интеграционно. Сейчас от него защищает *source-inspection* тест (проверка, что в строке фильтра есть `apad,atrim`) — сломается от любого рефакторинга и не ловит регрессий поведения ffmpeg.
   **Решение:** opt-in интеграционный тест (`@pytest.mark.integration`, включается в CI отдельным джобом): сгенерировать ffmpeg'ом синтетику (клик-трек + синус), прогнать `render_pieces`+`assemble_continuous`, измерить длину и лаг `gcc_phat` (функция уже в репо!) — asserts на <5 мс.
2. **Верификационный harness не в репо.** Вся отладка держалась на прото-скриптах gcc_phat (rendered vs camera), которые удалены. Это главный measurement-инструмент проекта — он должен жить в `tools/verify_sync.py` (см. §10.3).
3. **CI-матрица не соответствует реальности**: тестируются 3.11/3.12, разработка на 3.14, `requires-python >= 3.10` не проверяется. Добавить 3.10 и 3.14.
4. **Зависимости не запинены** (`>=` без верхних границ, lock-файла нет) — CI и установки пользователей будут ломаться мажорами faster-whisper/PyQt6/numpy. Минимум — constraints-файл, лучше `uv lock`/`pip-tools`.
5. Непокрыты: `media.probe` (парсинг ffprobe JSON можно тестировать на фикстурах без ffmpeg), `transcript_export.build_sentence_groups` частично, `cli`-парсер overrides (`test_cli` есть, но не все флаги), `export._media_src`/URI-энкодинг (roadmap прямо просил тесты на percent-encoding — их нет).

## 9. Зависимости и упаковка

1. **`pydub` и `ffmpeg-python` не используются нигде** — убрать из [requirements.txt](requirements.txt) и из hiddenimports [whispersync.spec](whispersync.spec). Меньше зависимостей = проще установка (pydub тянет aiff/audioop-боль на 3.13+).
2. **`rich` используется в рантайме CLI** ([cli.py:16-21](whispersync/cli.py#L16-L21)) с фолбэком, но лежит только в requirements-dev. Либо перенести в requirements.txt, либо убрать импорт (фолбэк-print и так есть).
3. `tqdm` в dev-requirements не используется вовсе.
4. **PyInstaller spec не собирался давно**: нет `datas` для `faster_whisper` assets (мел-фильтры), нет hook на `ctranslate2` shared libs; `pydub`/`ffmpeg` — призраки. Если дистрибуция актуальна — нужен CI-джоб сборки.
5. `pyproject.toml` не содержит `[project.dependencies]` — `pip install .` даст пакет без зависимостей; entry point `whispersync-gui = "main:main"` указывает на модуль `main` вне пакета — при установке колесом сломается (модуль `main.py` не попадает в дистрибутив без `py-modules`).
6. `.sep-venv` лежит в корне репо и подхватывается только по захардкоженному пути `repo_root/.sep-venv` ([separation.py:28-40](whispersync/engine/separation.py#L28-L40)) — в PyInstaller-сборке `Path(__file__).parents[2]` укажет внутрь бандла. Путь к сепаратор-venv должен быть конфигурируемым.

## 10. Предложения: новые фичи

Отсортированы по соотношению «ценность / стоимость».

### 10.1. Auto-strategy (высокая ценность, дёшево)
После выравнивания уже известны residual, распределение якорей и разброс локальных K. Правило: residual мал и K стабилен → стратегия 1; сильная нелинейность → 2; иначе — 4. Выводить рекомендацию в warnings/GUI («Strategy 1 would suffice: drift is linear, residual 8 ms») или добавить `--strategy auto`. Roadmap Backlog это предусматривал.

### 10.2. Акустический fallback при бедном транскрипте (высокая ценность, средне)
Главный отказной сценарий — мало якорей (музыка, шум, тишина, иностранный язык). Весь инструментарий уже в репо: `gcc_phat` + coarse-скан окнами по клипу даст offset/K вообще без слов — «Strategy 0 / PluralEyes-режим». Это превращает WhisperSync из «работает на речи» в «работает всегда, на речи — точнее».

### 10.3. `tools/verify_sync.py` — самопроверка результата (высокая ценность, дёшево)
CLI-утилита: берёт готовые `*_voice.wav` + исходные видео, меряет realized lag по сетке gcc_phat, печатает median/p90/max по клипу и красит проблемные регионы. Опция `--verify` в основном CLI после рендера: «DJI_0836: median 11 ms, max 65 ms @ 57%». Это тот harness, которым проект уже отлаживался — он должен стать фичей для пользователя и регрессионным инструментом для разработки.

### 10.4. Per-camera lip-sync калибровка (важно для основного пользователя)
Память проекта фиксирует: постоянная задержка «микрофон↔губы» внутри камеры (DJI Pocket) не ловится ничем акустическим. Добавить `camera_av_offset_ms` (конфиг per-camera / GUI поле) — одно число, прибавляемое к позиции аудиоклипов. Однократная калибровка хлопком.

### 10.5. Seam-snap-to-silence (это фикс §2.2, но по сути фича качества)
Границы кусков — только в меж-словные зазоры. Устраняет заикание без искажения синхронизации.

### 10.6. Экспорт в другие NLE (средняя ценность, средне)
OpenTimelineIO как промежуточный формат даёт Premiere/Resolve-нативный импорт почти бесплатно; EDL — тривиален. Сейчас FCPXML в Resolve работает, но Premiere-пользователи мимо кассы.

### 10.7. «Render synced master WAV» (дёшево)
Один WAV на всю длину таймлайна (все `_voice` куски на своих местах) — для пользователей без NLE-импорта (аудио-подкасты, черновой просмотр). `assemble_clip` из мёртвого кода — почти готовая реализация.

### 10.8. GUI: очередь заданий и «повторить с другой стратегией»
Транскрипции кэшированы — пересинк другой стратегией занимает лишь время рендера. Кнопка «Re-run with Strategy N» после первого прогона — очень дешёвый и очень заметный UX-выигрыш.

### 10.9. Диаризация/мульти-спикер (дорого, потом)
Для нескольких петличек `--recorder-mode all` уже кладёт дорожки отдельно; диаризация позволила бы автоматически выбирать «чей» рекордер главный в каждой фразе.
